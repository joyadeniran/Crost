Architecture upgrade

Orc is not a department — he is the Chief of Staff

The previous spec treated Orc as a one-shot decomposition agent: plan → approve → fan-out → aggregate. That was wrong. Orc is a persistent supervisory agent who stays alive for the full lifecycle of every goal. He reports everything to the founder. He never stops working until the goal is resolved.

What Orc is — the five responsibilities
1

Goal intake and decomposition

Receives the founder's goal. Queries all departments for current state data before drafting anything. Produces a structured JSON plan with a mandatory risk note. Shows the plan to the founder for selective approval. This is the same as before — but it is only the beginning of Orc's work, not all of it.

2

Real-time supervision of executing departments

After fan-out, Orc maintains a live view of every department's task state. He polls department status via the Onyx API and Supabase Realtime. If a department stalls, errors, or hits an unexpected blocker, Orc surfaces it immediately — he does not wait for the founder to notice. This is the persistent loop the previous spec was missing.

3

Workload adjustment and rebalancing

If Marketing finishes early and Sales is blocked, Orc can reassign, reprioritise, or modify task parameters — within the bounds the founder approved. He can pause a department, redirect its effort, or escalate to the founder for a decision. Like a real chief of staff who keeps the team moving when things change mid-execution.

4

On-demand status reporting

Founder can ask Orc at any time: "Where are we on December push?" Orc pulls live task state from all departments, synthesises it into a human-readable summary, and responds in the War Room. No need to check each department card individually. This is the chief of staff briefing — available on demand, always current.

5

Goal closure and post-mortem memo

When all tasks resolve (completed, rejected, or expired), Orc writes a structured closing memo to company_memos: what was planned, what executed, what was rejected, what the outcomes were. This becomes the institutional memory of every goal. Future goals query this memo as context — Orc learns from history.

How this maps to Claude's subagent architecture

Anthropic's multi-agent pattern has the orchestrator maintain context across the full task lifecycle — it can inspect subagent state, redirect them mid-execution, and is responsible for the coherent outcome, not just the initial decomposition. Orc is exactly this pattern applied to a business operating system.

Key implication: Orc is not a one-shot Onyx persona call. He is a stateful agent with a persistent session for the duration of each active goal. In Onyx terms: one long-running chat session per goal, with Orc continuously calling Internal Search, querying department status via custom MCP actions, and writing memos as tool outputs.
What changes in the architecture

Before (one-shot)

Orc produces plan → approval → fan-out → wait → aggregate at end. One prompt in, one plan out. Orc goes idle after fan-out.

After (persistent loop)

Orc stays alive for the goal lifecycle. Polls, supervises, rebalances, reports. Closes goal with a post-mortem memo. One active Orc session per active goal.

New schema additions for persistent Orc
  • +goals table gains: orc_session_id (Onyx chat session ID for persistent supervision), last_status_check (timestamp), supervision_interval_seconds (default 30)
  • +goal_tasks table — individual task rows per goal: task_id, dept_slug, status, assigned_at, completed_at, orc_notes (JSONB rebalancing history)
  • +New event_type values: orc_status_check, orc_rebalance, orc_escalation, goal_closed, goal_post_mortem_written
  • +New approval type: orc_rebalance — when Orc wants to modify a department's task mid-execution, this goes through the approval feed like everything else
Orc's runtime loop (pseudocode)
while goal.status === 'executing':
  task_states = poll_all_department_statuses(goal.tasks)
  
  for task in task_states:
    if task.status === 'error':
      escalate_to_founder(task, reason='error')
    if task.is_stalled(threshold=5min):
      escalate_to_founder(task, reason='stalled')
    if task.status === 'completed':
      check_for_rebalance_opportunity(task, other_tasks)
  
  if founder_asks_for_status():
    return synthesise_status_brief(task_states)
  
  sleep(supervision_interval_seconds)

on goal_resolved:
  write_post_mortem_memo(goal, task_states)
  update goal.status = 'completed'
Dedicated training

Orc needs a training corpus, not just a system prompt

A system prompt alone makes Orc a smart generalist. Dedicated training makes him a domain expert in this founder's business. The difference is whether Orc knows the founder's retailer data, their seasonal patterns, their past goals, their tolerance for risk — or whether he's reasoning from scratch every time.

Layer 1 — System prompt (always injected)
  • CCrost Constitution (9 clauses including Orc's mandatory risk note clause)
  • RChief of Staff role definition — what Orc is, what he is not, his relationship to the founder and departments
  • LLocal identity — founder name, city, business, stage, risk tolerance
  • DDepartment registry — all active departments, their capabilities, their current status
  • OOutput format rules — JSON for plans, structured status briefs, memo format for post-mortems
Layer 2 — Knowledge corpus (indexed in Onyx / Vespa)

Orc's Onyx persona subscribes to specific knowledge corpora that give him business context. This is where Onyx's RAG capability is fully leveraged for the first time.

  • +company_memos corpus — every memo ever written by any department. Orc reads all of them. This is his institutional memory. When planning a December sales push, he already knows what the October one found.
  • +goals corpus — every past goal, plan, and post-mortem. Orc learns from what worked and what didn't. "Last time we tried aggressive outreach, 3 retailers unsubscribed" is context Orc should have.
  • +business context documents — uploaded by the founder at onboarding and updated over time: product catalogue, pricing, key customers, supplier contacts, business rules ("never message a retailer twice in 24 hours"). Stored as indexed files in Onyx.
  • +approval history — what the founder has approved vs rejected. Orc learns the founder's preferences over time. If the founder always rejects Ops tasks during planning, Orc starts deprioritising them automatically.
Layer 3 — Live context (injected at runtime)
  • →Memo brief — high-priority memos written in the last 24 hours that Orc hasn't read
  • →Active goal states — what tasks are currently running, their status, any pending approvals
  • →Department health — current status of each department (idle / running / awaiting_approval / error)
  • →env_mode — local or cloud, so Orc assigns models correctly to each task
Layer 4 — Business rules document (founder-editable)

A dedicated "Business Rules" document in Onyx that the founder can edit directly. These are injected into Orc's context on every call and cascade to workers via the memo system.

Format: plain language rules, one per line. "Never message a retailer twice within 24 hours." "All spend above N50,000 requires founder approval regardless of risk_tolerance setting." "Marketing drafts must be reviewed by founder before sending during Ramadan period." Orc treats these as constitutional-level constraints for planning.

The three that change the MVP the most
Implement now

Celery for Orc's supervision loop

The persistent supervision loop cannot run in the frontend or as a polling interval in a Next.js route — it will die when the browser tab closes. Onyx already has Celery with high/medium/low priority queues, retry logic, heartbeat checking, and Redis coordination. Orc's supervision loop is a Celery periodic task. This is free infrastructure already in the stack.

Implementation: create a Celery beat task `orc_supervision_tick` that runs every 30 seconds per active goal. Task polls department status, checks for stalls, escalates if needed. No new infrastructure required — Onyx's existing worker handles it.
Implement now

Supabase connector replaces manual query tool

The current spec builds a custom supabase_query tool. Onyx already has a connector framework that can index and query Supabase data natively, with incremental sync and access control. Workers don't need a custom tool — they use Onyx's Internal Search action pointed at the Supabase connector. The data is indexed, searchable, and RAG-queryable.

Set up the Supabase connector in Onyx admin pointing to the founder's business database (retailers, transactions, pipeline). Workers query via Internal Search. The connector handles auth, sync, and permissions automatically.
Implement Phase 2

MCP for Orc → approval feed integration

Currently Orc signals approval requests by returning a string that onyx-client.ts parses with regex. That is fragile. MCP lets Orc call a structured action that directly creates an approval_queue entry. The approval is a typed function call, not a parsed string. This is more reliable, more testable, and follows Anthropic's own MCP guidance for agentic tool use.

Expose Crost's approval API as an MCP server. Orc's persona gets the `request_approval` MCP action. When Orc needs approval, it calls the action directly — structured parameters, no regex parsing, full type safety.
